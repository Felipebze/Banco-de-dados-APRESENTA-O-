Fundamentos

O que é, afinal, um Banco de Dados?

Um banco de dados é o núcleo responsável por armazenar, organizar e disponibilizar informações com rapidez, segurança e confiabilidade — tornando possível o funcionamento de aplicações modernas. Pense nele como a memória estrutural de qualquer produto digital: invisível, mas indispensável.

Empresas

Bancos guardam catálogos de produtos, estoque, transações financeiras e histórico de clientes em tempo real.

Aplicativos

Toda mensagem, curtida, foto ou localização que você envia é lida e escrita em um banco de dados.

Governos

CPFs, processos judiciais, registros civis e votos são armazenados em bancos com altíssima exigência de integridade.

Uma linha do tempo resumida

Do papel aos vetores semânticos: como os dados evoluíram.

1960s — Modelo Hierárquico e em Rede

Sistemas como o IBM IMS organizavam dados em árvores rígidas, feitos sob medida para cada aplicação — difíceis de mudar e de consultar livremente.

1970 — O Modelo Relacional

Edgar F. Codd, pesquisador da IBM, publica o artigo que propõe organizar dados em tabelas relacionadas por chaves — a base de tudo que viria depois, incluindo o SQL.

1979 — Primeiro SGBD Relacional Comercial

A Relational Software Inc. (futura Oracle) lança o primeiro banco relacional comercial baseado em SQL.

1995–2009 — Era Open Source e Web

MySQL e PostgreSQL se popularizam junto com a explosão da internet e de aplicações web dinâmicas.

2009 — Ascensão do NoSQL

MongoDB, Cassandra e Redis surgem para lidar com o volume e a velocidade de dados de redes sociais e big data.

2020s — Bancos Vetoriais e IA

Com a explosão dos LLMs, bancos de dados passam a armazenar vetores (embeddings) para permitir busca por significado, não apenas por texto exato.

Curiosidade: o termo "banco de dados" (database) só se popularizou nos anos 1960, mas o conceito de organizar informação em registros estruturados remonta a censos e catálogos de bibliotecas de séculos atrás.

Agora que entendemos o papel dos bancos de dados, vamos conhecer os sistemas responsáveis por gerenciá-los.

Conceito

O que é um SGBD?

Um Sistema Gerenciador de Banco de Dados (SGBD) é o software que fica entre a aplicação e os dados brutos. Ele não é o banco em si — é o "gerente" que controla como os dados são gravados, lidos, protegidos e organizados.

Segurança e controle de acesso

O SGBD decide quem pode ler, alterar ou apagar cada informação, através de usuários, papéis e permissões granulares.

Integridade dos dados

Garante que regras de negócio (como "um CPF não pode se repetir") sejam sempre respeitadas, mesmo com milhares de operações simultâneas.

Desempenho

Usa índices, cache e otimizadores de consulta para responder em milissegundos, mesmo em tabelas com bilhões de linhas.

Concorrência e transações

Permite que milhares de pessoas leiam e escrevam ao mesmo tempo sem corromper os dados, através de transações ACID.

Vetorização: quando dados viram números

Para uma IA "entender" significado, o texto precisa virar matemática. É isso que a vetorização faz.

Palavras
Embeddings
Vetores
Espaço Vetorial
Similaridade

O que é um Embedding?

Um embedding é uma lista de números (um vetor) que representa o significado de um texto, imagem ou som em um espaço matemático multidimensional. Frases com significados parecidos geram vetores próximos entre si.

Do texto ao vetor

Um modelo de embedding lê a frase "gato correndo no parque" e converte cada pedaço de significado em coordenadas — algo como [0.12, -0.87, 0.34, ...], geralmente com centenas ou milhares de dimensões.

Espaço vetorial semântico (simplificado em 2D)

Palavras com significados próximos (ex: "cachorro" e "gato") aparecem mais perto umas das outras. Palavras sem relação (ex: "banco de dados" e "praia") ficam distantes.

Distância Euclidiana

Mede a distância "em linha reta" entre dois vetores no espaço — quanto menor, mais parecidos.

Similaridade do Cosseno

Mede o ângulo entre dois vetores, ignorando magnitude — a métrica mais usada em busca semântica de texto.

KNN vs. ANN

KNN busca os vizinhos mais próximos de forma exata; ANN (Approximate Nearest Neighbor) troca um pouco de precisão por muito mais velocidade em bases gigantes.

Na prática: quando você pesquisa "roupa para treinar" numa loja com busca semântica, o banco vetorial não procura essas palavras exatas — ele encontra produtos como "camiseta dry-fit" e "legging esportiva", porque seus vetores estão próximos no espaço semântico.
Convergência

Bancos de Dados e Inteligência Artificial

LLMs (Large Language Models) não "sabem" fatos novos ou privados por padrão — é aí que a busca vetorial entra em cena, através de uma arquitetura chamada RAG.

Como um LLM funciona (resumo)

Um LLM é treinado em enormes volumes de texto para prever a próxima palavra mais provável, dado um contexto — é essa capacidade estatística que gera respostas coerentes.

O que é RAG?

Retrieval-Augmented Generation: antes de responder, o sistema busca informações relevantes em um banco vetorial e as injeta no contexto do LLM — assim ele "consulta" dados atualizados ou privados em vez de confiar só na memória do treinamento.

Resposta
01Chunking — dividir o documento

Documentos longos são quebrados em pedaços menores (chunks), geralmente de algumas centenas de palavras, para que cada busca traga apenas o trecho relevante — não o documento inteiro.

02Embedding — transformar em vetores

Cada chunk passa por um modelo de embedding e vira um vetor numérico, capturando seu significado semântico.

03Armazenamento — banco vetorial

Esses vetores são salvos em um banco preparado para busca por similaridade, como o pgvector (extensão do PostgreSQL), Pinecone ou Qdrant.

04Busca semântica — encontrar o que importa

Quando o usuário faz uma pergunta, ela também vira um vetor, e o banco encontra os chunks com vetores mais próximos — ou seja, mais relevantes ao significado da pergunta.

05Geração — o LLM responde com contexto

Os trechos mais relevantes são enviados junto com a pergunta ao LLM, que gera uma resposta fundamentada nesses dados — reduzindo alucinações e permitindo respostas sobre dados privados ou recentes.

Fechamento

De tabelas a vetores semânticos

Bancos de dados evoluíram de simples arquivos organizados para sistemas capazes de entender significado — mas o princípio continua o mesmo: guardar informação de forma que ela possa ser encontrada rapidamente, quando for necessária.

Relacional

Estrutura, integridade e décadas de maturidade — a base de praticamente todo sistema crítico.

NoSQL

Flexibilidade e escala horizontal para dados que mudam de forma constante.

Vetorial

A nova fronteira: dados organizados por significado, não apenas por valor exato.

Modelagem

Banco de Dados Relacional

O modelo relacional organiza dados em tabelas — como planilhas conectadas entre si por identificadores únicos.

Tabela

Uma coleção de dados sobre um mesmo assunto — ex: "clientes", "pedidos", "processos".

Chave Primária

Identifica de forma única cada linha de uma tabela — ex: o CPF de um cliente.

Chave Estrangeira

Aponta para a chave primária de outra tabela, criando o relacionamento entre elas.

1 : 1

Uma pessoa tem um único CPF, e um CPF pertence a uma única pessoa.

1 : N

Um cliente pode ter vários pedidos, mas cada pedido pertence a um único cliente.

N : N

Um aluno cursa várias disciplinas, e cada disciplina tem vários alunos — resolvido com uma tabela intermediária.

Modelagem

Banco de Dados Não Relacional

Bancos NoSQL abandonam a rigidez das tabelas para ganhar flexibilidade e escala horizontal.

Documentos

Dados salvos como documentos JSON/BSON flexíveis. Ex: MongoDB. Ideal para catálogos com atributos variáveis.

Chave-Valor

Cada dado é acessado por uma chave única, como um dicionário gigante. Ex: Redis. Ideal para cache e sessões.

Colunar

Dados organizados por colunas em vez de linhas. Ex: Cassandra. Ideal para grandes volumes analíticos e séries temporais.

Grafos

Dados como nós e conexões. Ex: Neo4j. Ideal para redes sociais e sistemas de recomendação.

Quando usar NoSQL

  • O formato dos dados muda com frequência
  • Escala horizontal massiva é prioridade
  • Baixa latência de leitura é crítica (cache)

Quando evitar NoSQL

  • Relacionamentos complexos entre entidades
  • Transações financeiras que exigem consistência forte
  • Relatórios com muitos joins entre tabelas
